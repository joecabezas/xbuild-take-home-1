Architecture decisions made during planning:

- Each pipeline worker runs in its own Docker container (validator, normalizer, matcher, pricer, assembler)
- Workers communicate via RabbitMQ (chosen over Redis for AMQP semantics, dead-lettering, and management UI)
- The gateway uses a request-reply pattern over RabbitMQ: publishes to "validate" queue, waits on a
  per-job reply queue for the completed PipelineContext
- SQLite is shared via a named Docker volume, mounted only into gateway and assembler (the only two
  containers that touch the DB)
- Proposals are append-only / immutable: each generate-proposal call creates a new version row,
  old proposals remain forever retrievable — versioning is built-in by design, not a future concern
- A shared/ Python package is installed into every container, containing PipelineContext,
  queue_client, models, db, and repos — single source of truth, no duplication
- Photos are strings throughout (no binary storage anywhere)
- The RabbitMQ management UI at :15672 lets the interviewer watch messages flow through queues
